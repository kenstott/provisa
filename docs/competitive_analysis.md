# Competitive Analysis

Products: Provisa, Neo4j, Apollo GraphOS, Starburst, Hasura

---

| Feature / Feature Set | Provisa | Neo4j | Apollo GraphOS | Starburst | Hasura |
|---|---|---|---|---|---|
| **Data Source Coverage** | Exceeds all. 30+ source types: RDBMS, NoSQL, graph, streaming (Kafka), files (CSV, Parquet, SQLite), REST/GraphQL/gRPC APIs, WebSocket, Google Sheets, object storage. Every source is a first-class citizen — all features apply uniformly regardless of source type. | Graph database only. Data from other sources must be ETL'd in. No native support for files, streams, or APIs as queryable sources. | Does not connect to data sources directly. Federates GraphQL services that teams have already built. Source diversity is the responsibility of individual subgraph authors. | Exceeds Neo4j, Apollo, Hasura. Strong analytics connector ecosystem: RDBMS, data lakes, object storage, Hive, Iceberg. Weaker on operational sources — no native files-as-tables, limited API/stream support. | Primarily relational databases. Remote schemas and actions extend coverage but without governance parity. NoSQL, files, and streams are not first-class — actions are code the team must write and maintain. |
| **All Sources First-Class** | Exceeds all. RLS, column masking, query approval, caching, MV optimization, NL query, relationship traversal, and per-role schema visibility apply identically across every source type. No feature gaps by source. Remote GraphQL/SDL sources follow the same pattern: the remote SDL is the introspection bootstrap — the floor, not the ceiling. After registration, the source is a full semantic layer citizen: governance (RLS, masking, domain permissions, query approval) layered on top, semantic relationships extended to local and other remote sources, resolution overridden via explicit FK/PK declarations (replacing remote-native joins with Trino-cached ones), columns aliased, native filter columns and required args applied, and naming conventions applied independently of what the SDL exposes. The remote endpoint has no knowledge of any of this. | N/A — single source type. | Governance is per-subgraph, implemented by the subgraph author. No platform-level enforcement across sources. | Governance (row filters, column masking) applies at query level but not uniformly across all connector types. | Features vary by source type. RLS is strong on native database sources; behavior on remote schemas and actions is inconsistent. v2 column redaction (nulling unauthorized fields) is available on supported databases (Postgres, Citus, MSSQL, BigQuery) but not on MySQL, MongoDB, Snowflake, or others — not enterprise-only, but source-dependent. DDN has no native column masking; masking requires a custom Lambda connector. |
| **Query Languages** | Exceeds all. SQL and GraphQL are fully first-class query interfaces. Cypher is supported for core graph use cases (path queries, variable-length traversal, connected entity lookup) with governance (RLS, masking, approval) applied. Cypher coverage does not extend to GDS analytics (PageRank, community detection) or Neo4j external libraries/plugins (APOC, etc.) — those remain Neo4j-native strengths. | Cypher (native). Basic SQL via JDBC driver. No GraphQL without third-party library. | GraphQL only. No SQL, no Cypher. | SQL only (Trino dialect). No GraphQL, no Cypher. | GraphQL only. No SQL, no Cypher. |
| **Natural Language Query** | Exceeds all. Claude-powered NL→SQL/Cypher/GraphQL pipeline with interactive validation loop. Targets all three query languages depending on source type. | Not available. | Not available. | Not available. | Not available. PromptQL (separate product) offers NL query but requires full platform migration to DDN. |
| **API Output Formats** | Exceeds Apollo, Starburst, Hasura. Exceeds Neo4j on general-purpose API formats. JSON, NDJSON, CSV, Parquet, Apache Arrow — all with HTTP content negotiation. gRPC (auto-generated proto per role), Arrow Flight (unbounded columnar streaming), JDBC, Python client (pandas, pyarrow, SQLAlchemy, ADBC, DB-API 2.0), REST/JSON:API, WebSocket/SSE subscriptions, Kafka sink. Bolt protocol not supported — Bolt is graph-native and not applicable outside a Neo4j-compatible server. | JDBC, Bolt protocol, HTTP REST. Limited to Neo4j-native formats. No Arrow, no gRPC, no content negotiation. | GraphQL over HTTP only. No Arrow, no gRPC, no JDBC, no columnar output. | SQL/JDBC, Arrow Flight (via Trino). No GraphQL, no gRPC, no REST/JSON:API. Strong for analytics consumers, limited for application consumers. | GraphQL over HTTP, WebSocket subscriptions. No Arrow, no gRPC, no JDBC, no columnar output. |
| **Federation / Query Execution** | Exceeds Neo4j, Apollo, Hasura. Trino-compatible federation. Single-source queries bypass Trino entirely (sub-100ms target). Smart routing is automatic. BYO Trino supported. RAM budget auto-scales local worker count. Remote GraphQL sources participate fully in federation: FK/PK overrides replace remote-native resolver joins with Trino-cached resolution, enabling cross-source joins between a remote GraphQL endpoint and any other Provisa source — something the remote endpoint itself cannot express. | No federation. Single graph store. | Composes GraphQL services — not data federation. Requires pre-existing GraphQL subgraphs. No cross-source query execution. | Best-in-class Trino federation for analytics workloads. Mature CBO with statistics collection. Exceeds Provisa on raw query scale and CBO maturity. | Remote joins enable limited cross-source queries. No true federation engine — joins are resolver-chain based, not SQL-level. |
| **Query Optimization** | Exceeds Neo4j, Apollo, Hasura. No Cost-Based Optimizer — Provisa trades CBO breadth for targeted high-value optimizations: hot table value CTEs (frequently referenced lookup values pre-materialized and injected automatically), automated MV rewrite (transparent join pattern matching), manual MV optimization (steward-controlled with scheduled refresh), cross-datasource semantic relationships inform the optimizer. For workloads dominated by repeated lookup patterns and known join shapes, these optimizations often deliver more practical gain than a general-purpose CBO. Starburst exceeds Provisa on raw CBO depth for complex ad-hoc analytical queries. | Mature Cypher query planner with cost-based optimization, index-backed lookups, and execution plan tuning — likely exceeds Provisa for native graph query optimization. No equivalent to Provisa's cross-source MV rewrite, hot CTE layer, or semantic relationship hints, as those are multi-source concepts that don't apply to a single-store graph database. | No query optimization. | Cost-Based Optimizer (CBO) with mature statistics pipeline. Exceeds Provisa on CBO depth. No MV auto-rewrite, no hot CTE layer. | No query optimization beyond basic resolver execution. |
| **Large Result Delivery** | Exceeds all. Threshold-based automatic S3 redirect for oversized result sets. Federated write path delivers Parquet/ORC directly to object storage — data never passes through the Provisa server. Multi-root query support (some fields inline, some redirected). Transparent to caller. | Manual export only. APOC procedures (apoc.export.csv, apoc.export.json, etc.) can write results to disk or S3, but require explicit invocation — not automatic or threshold-based. | No equivalent — Apollo is a federation layer and does not execute queries against data directly. | Manual write only. CTAS writes query results to object storage, and INSERT INTO supports pipeline-style unloading. Requires explicit query construction — not automatic or threshold-based. | No equivalent. |
| **Row-Level Security** | Exceeds Apollo, Neo4j. Per-table, per-role WHERE clause injection at SQL level. Inherited roles with recursive RLS inheritance. Applied uniformly across SQL, GraphQL, and Cypher. Security-partitioned cache keys ensure cached results never cross role boundaries. | Role-based access only. No row-level filtering equivalent to SQL RLS. | No platform-level RLS. Each subgraph author implements their own filtering. No enforcement mechanism across subgraphs. | Row filters at query level. Strong for SQL workloads. Not applied across all connector types uniformly. | Strong RLS on native database tables. Inconsistent on remote schemas and actions. |
| **Column Masking** | Exceeds Apollo, Neo4j. Per-column per-role masking injected at SQL level. Mask types: regex (REGEXP_REPLACE), constant (literal/NULL/MAX/MIN), truncate (DATE_TRUNC). Applied uniformly across all source types and query languages. | No column masking. | No column masking. | Column masking available. Comparable to Provisa for SQL sources. | v2: column-level permission redaction nulls unauthorized fields via inherited roles — this is access-based hiding, not data transformation. Not enterprise-only but database-dependent (unsupported on MySQL, MongoDB, Snowflake, and others). DDN: no native column masking. TypePermissions hide fields from the schema entirely. Actual masking (regex, hashing, truncation) requires a custom Lambda connector or plugin that the team must write and maintain. |
| **Per-Role Schema Visibility** | Exceeds all. Different roles see completely different GraphQL schemas with field-level visibility. Unauthorized tables and columns are absent from SDL — not hidden, structurally unreachable. Per-role gRPC proto generation. | Role-based access but no per-role schema generation. | Contract schemas provide some differentiation but not field-level per-role SDL generation. | No per-role schema — SQL returns what the query requests subject to filters. | Per-role schema visibility on native tables. Less consistent on remote schemas. |
| **Domain Permission Model** | Unique — no competitor has this. Queries execute within a domain context bounded by the domain's own sources plus everything reachable via approved relationship traversal. Data outside the traversal graph is structurally unreachable — not denied, non-existent from the caller's perspective. Relationships carry sensitivity tiers (domain/public/restricted); virtual metadata columns visible by default, everything else requires access request. Composition-based, not subtraction-based. | No domain model. Role-based access only. | No domain model. Schema ownership is organizational, not enforced at query time. | No domain model. Access control is query-level, not domain-scoped. | No domain model. Permission model is table and column based, not domain-traversal based. |
| **Query Approval Workflow** | Exceeds all. Governed query registry with full approval lifecycle: submission, review, approval, deprecation with replacement pointer. Captures business purpose, data sensitivity, refresh frequency, expected row count. Stable IDs enable CDN/proxy caching via GET. Ceiling enforcement — clients may restrict within approved query but cannot exceed it. Schema changes flag affected registry entries for re-review. | No query approval workflow. | Schema checks and linting in GraphOS. No data-level query approval, no ceiling enforcement. | No query approval workflow. | No query approval workflow. |
| **Caching** | Exceeds all. Redis-backed query result cache. Security-partitioned cache keys (role + RLS context). Steward-controlled TTL per approved query. Cache invalidation on registration changes and mutations. `X-Provisa-Cache: HIT|MISS` response headers. Apollo APQ with Redis-backed hash→query cache. | No result caching. | Operation caching available. Not security-partitioned by role/RLS context. | No governed result cache. | Basic query caching available. Not security-partitioned. |
| **Real-Time Subscriptions** | Exceeds all. WebSocket, SSE, and Kafka sink output. Backends: PostgreSQL native, MongoDB native, Debezium CDC, polling. Subscriptions work across all supported source types — not limited to native databases. Governed — same RLS and masking applied to subscription events. Kafka sink is unique: no competitor routes governed subscription events directly to a Kafka topic. | Streaming via Bolt protocol for graph change events. Not a general subscription platform. | Limited subscription support. Not a primary feature. | No subscription/streaming support. Analytics-focused. | Strong subscriptions via WebSocket on native database tables. No Kafka sink. Coverage does not extend uniformly to remote schemas and actions. |
| **Graph Query / Cypher** | Exceeds Neo4j for governed use cases. Cypher is a supported query interface — full governance stack applied (RLS, masking, domain permissions, query approval). Covers connected entity lookup, bounded path queries, org hierarchies, and recommendations. Limitations: unbounded path queries not supported (also an anti-pattern in Neo4j in practice), GDS analytics (PageRank, community detection), and Neo4j external plugins (APOC, etc.) are not supported. | Native graph database. Best-in-class for deep traversal and GDS analytics (PageRank, community detection, centrality, fraud detection pattern matching). No governance layer — no column masking, no query approval, no per-role schema, no domain permission model. | No graph query support. | No graph query support. | No graph query support. |
| **AI Relationship Discovery** | Unique — no competitor has this. Claude-powered foreign key candidate suggestion across sources. Scope control: per-table, per-domain, cross-domain. Candidate lifecycle: suggested, accepted, rejected, expired. Sample data validation for high-confidence candidates. | No equivalent. | No equivalent. | No equivalent. | No equivalent. |
| **Column Semantic Metadata** | Exceeds all. Columns carry semantic tags: metric, dimension, identifier, timestamp, ops domain. Configurable inference (auto-tag numeric columns as metrics, low-cardinality as dimensions, FK candidates as identifiers). Enriches NL query context, JDBC catalog mode, and data discovery tools. | No equivalent. | No equivalent. | No equivalent — column metadata is schema-level only. | No equivalent. |
| **Vector Search** | Not yet available. | Vector index support added in v5+, within Neo4j store only. No cross-source vector search, no fallback materialization, no declarative generation, no governed pipeline. | No vector search capability. | No vector search capability. Trino has no native cosine similarity function or vector index support. | No governed vector search. PromptQL includes some vector capability but tied to Hasura's AI product, not the governed data API. |
| **No-Code Configuration** | Exceeds Apollo, Starburst, Neo4j. Entire platform configured via YAML. Adding a data source, registering a table, defining relationships, configuring RLS/masking, declaring embedding columns — all config, no code, no deployment. Config reload without restart. | Requires code for application integration. No config-driven governance. | Requires writing and deploying subgraph services. Every new source is an engineering project. | Requires SQL and connector configuration. No config-driven API layer. | Config-driven for database sources. Comparable to Provisa for RDBMS. Requires code (actions) for non-database sources. |
| **Deployment Flexibility** | SaaS: not available — disadvantage for buyers who want zero ops. On-prem: exceeds all. Laptop (DMG/AppImage, one-click), single VM (Docker Compose), HA VM cluster (multiple instances + shared Redis + config DB, no orchestration required), Kubernetes (Helm, HPA). Air-gapped: fully supported, no internet dependency. Same product, same config, same governance across all tiers. BYO Trino supported. | SaaS: AuraDB (managed cloud). On-prem: VMs, Docker, Kubernetes. Laptop: Neo4j Desktop available. No air-gap-specific features for AI/LLM capabilities. | SaaS: GraphOS (managed platform, schema registry, observability). On-prem: Apollo Router is self-hostable, but the GraphOS management plane is SaaS-only — no fully on-prem option. No air-gap. | SaaS: Galaxy (fully managed). On-prem: SEP (Starburst Enterprise Platform), Kubernetes-only, separate pricing tier, heavyweight ops burden. No VM or laptop deployment. Air-gap: SEP viable but operationally heavy. | SaaS: Hasura Cloud (v2), DDN Cloud (v3). On-prem: self-hosted Docker/Kubernetes for both v2 and DDN. No single-binary or laptop deployment. DDN self-hosted is viable but DDN Cloud is the primary supported path. |
| **Regulated Industry / Air-Gap** | Exceeds all. On-prem deployment on existing infrastructure (VMs, K8s, or single server). No cloud dependency, no phone-home. Local embedding models via Ollama/HuggingFace for fully air-gapped AI features. Per-tenant S3 bucket support for data residency requirements. | Deployable on-prem. No air-gap-specific features for AI/LLM capabilities. | Not viable for regulated industries — SaaS only. | Galaxy not viable — SaaS only. SEP viable but Kubernetes-only and heavyweight. | Hasura Cloud not viable for strict data residency. Self-hosted viable but DDN pushes toward cloud. |
| **Migration / Import Tools** | Has what others lack. Hasura v2 metadata importer (converts sources, relationships, permissions to Provisa config). DDN supergraph YAML converter. Apollo Federation v2 subgraph exposure (Provisa exposes itself as a subgraph). | No migration tooling for other platforms. | No migration tooling for other platforms. | No migration tooling for other platforms. | Native upgrade path from v2 to DDN requires full rewrite — not a migration, a replacement. No import from other platforms. |
| **Hasura v2 Upgrade Path** | Has what Hasura lacks. Provisa imports Hasura v2 metadata directly. Customers stranded by the v2→DDN breaking change have a clear migration path to Provisa with existing config preserved. | N/A | N/A | N/A | v2 is effectively end-of-life. DDN (v3) is a completely different architecture — new config format, new mental model, new deployment. No automated upgrade path. DDN is now primarily a feature within PromptQL, not a standalone governed data API product. |
| **SaaS / Managed Option** | Does not currently offer SaaS. On-prem and self-hosted only. Architecture supports multi-tenant SaaS (per-tenant config DB, Redis key namespacing, spill disabled per session, per-tenant S3 prefix) — not yet productized. Disadvantage for buyers who want zero ops. Advantage for regulated industries where SaaS is disqualified. | AuraDB (managed cloud). On-prem available. | SaaS only (GraphOS). Strong managed offering. | Galaxy (SaaS, strong managed offering). SEP (on-prem, Kubernetes). | Hasura Cloud (SaaS). Self-hosted available. DDN Cloud is the primary managed offering. |
